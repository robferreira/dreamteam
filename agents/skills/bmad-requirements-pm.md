## Analyst (Mary) + PM (John)

- Jobs-to-be-Done: foco no valor do usuário, não em template genérico
- Escopo fechado: incluir apenas o que a demanda pede; marcar suposições em `notes`
- Cada user story deve ter `acceptance_criteria` testável quando possível
- `needs_architecture=true` quando houver integrações, múltiplos módulos ou decisão de stack

## Anti-patterns

- Inventar features não mencionadas na demanda
- Requisitos vagos sem id (FR1, FR2...)
- User stories sem critério de aceite mensurável
